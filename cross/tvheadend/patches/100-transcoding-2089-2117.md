# Transcoding: filter pipeline rework for ffmpeg 6–8 + QSV/V4L2 M2M/NVDEC support

## Credit

This PR is a continuation of the work by @uknunknown in #2089 (*update filter to format required by ffmpeg8*) and #2117 (*transcoding update to filter2*), who unfortunately no longer has time to pursue it. The first commit of this series contains their work and is committed with them as the git author, so full credit is preserved in the history. Thank you @uknunknown for the enormous groundwork — none of this would exist without it.

## What this contains

- **The original rework from #2089/#2117**: migration of the filter initialization to `avfilter_graph_alloc_filter()` + `avfilter_init_str()` (required for attaching `hw_device_ctx` as of ffmpeg 6+), split of the ENCODE/DECODE open phases, hwaccel dispatch by `AVHWDeviceType`, new QSV and V4L2 M2M (Raspberry Pi 4+) hardware acceleration, nvenc aligned with the vaapi option model (qp, maxrate, scale factors, P1–P7 presets), SAR carried through the parsers/packets/muxers, and PTS/DTS/duration rescaling when a filter changes the frame rate.
- **Follow-up fixes from debugging real Intel VAAPI/QSV streams**: encode-error recovery (stops the infinite loop on `VA_STATUS_ERROR_ENCODING_ERROR`), no more `level=0` sent to VAAPI encoders, shared-transcoder audio loss on late join, empty hvcC on small HEVC SEI, QSV B-frame pinning, QSV device caching, the cosmetic MP4 muxer error on client disconnect, and a set of VAAPI/libvorbis/libopus default codec & stream profiles.
- **In-depth branch review fixes**: a NULL-codec crash in the VAAPI auto-detect path, migration of the old `hwaccel` setting to `decoder_hwaccel_type` (existing configs no longer silently lose hardware decoding), restored transcoder noop-packet handling in globalheaders, corrected 90 kHz packet durations when a filter changes the frame rate, MPEG-2 square-pixel SAR (`aspect_ratio_information=1` is a SAR, not a DAR), runtime software fallback when hardware decode init fails, no stream teardown on intentionally dropped frames (plus audio PTS-discontinuity resync), MPEG-2 display extension no longer overwriting the coded size, and global headers no longer gating stream start on the SAR.
- **ffmpeg 8.1 compatibility**: builds cleanly against ffmpeg 8.1.1; the codec-capability queries were migrated to `avcodec_get_supported_config()` (the last ffmpeg-8-deprecated API in the tree), with the old field reads kept behind version guards for older ffmpeg.
- **OMX (OpenMAX IL) removal + legacy path preserved**: OMX targeted the abandoned 32-bit Raspberry Pi stack and was the only codec never opting into the new pipeline (its modern replacement, V4L2 M2M, is added here). On builds against libavcodec ≤ 59 (ffmpeg ≤ 5.x, still accepted by `configure`), the encoder geometry/time_base setup the rework moved into filter negotiation is restored under a version guard, so older builds keep working.
- **Shared hwaccel helper deduplication**: the DRM device probe and the `hwdownload` filter builder, copy-pasted across the vaapi/qsv/nv backends, are factored into a shared `drmdev` unit and the common `hwaccels_download()` dispatch (addresses the SonarQube new-code duplication gate).
- **AAC default-profile fix**: the `webtv-*-aac-mp4`/`-matroska` profiles routed audio through `webtv-aac`, whose unset `profile` field loaded as 0 (AAC Main) and was injected into the encoder; the native `aac` encoder rejects Main, killing the whole audio transcode. `base_open()` now only emits a positive (explicitly chosen) profile.
- **WEBM container check**: the muxer now accepts VP9 and Opus (not just VP8/Vorbis) as valid WEBM streams, which it was already writing.
- **Cleanups**: SonarQube-flagged dead code, redundant casts/parentheses and one-declaration-per-line, plus a few Gemini correctness nits (encoder `time_base` computation, nvenc tuning constants, framerate-overflow clamp, stale-`errno` guard on short writes).

## Compatibility and testing

Build-validated against ffmpeg 7.1 and 8.1.1 (x64, VAAPI+QSV enabled, `-Werror` clean), with the fixes exercised on real DVB streams on Intel VAAPI/QSV hardware (including the AAC fix). An open question for the maintainers: with OMX gone, every codec uses the new filter pipeline, so the `has_support_for_filter2` transition flag could become a plain libavcodec version check — and if a minimum of ffmpeg ≥ 6 were acceptable, the entire legacy filter path could be dropped in a follow-up.
